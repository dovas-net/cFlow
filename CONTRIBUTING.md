# Contributing to flow

`flow` is a single-header C library for interactive node-graph editors in the terminal тАФ a
C / terminal analog of [xyflow](https://github.com/xyflow/xyflow) (React Flow). This guide is
the contributor workflow: how to edit, build, test, and submit changes so they pass CI. For
the per-function API reference see [`docs/API.md`](docs/API.md); for usage and the feature
list see [`README.md`](README.md).

---

## The one rule: `flow.h` is GENERATED тАФ never edit it by hand

`flow.h` is **not** source. It is a build product, amalgamated from the `src/*.h` modules by
[`tools/amalgamate.sh`](tools/amalgamate.sh). It is committed so that consumers can `curl` a
single header, but you regenerate it тАФ you do not hand-edit it. Its banner says so:

```
/* flow.h тАФ GENERATED by tools/amalgamate.sh. Do not edit; edit src/ instead. */
```

The workflow is always:

```sh
# 1. edit the real source
$EDITOR src/flow_render.h

# 2. regenerate the single header
make                    # runs tools/amalgamate.sh, then builds demos + examples
# or, header only:
sh tools/amalgamate.sh

# 3. commit BOTH the src/ change and the regenerated flow.h
git add src/flow_render.h flow.h
```

**A PR that edits `src/` but ships a stale `flow.h` fails CI.** The `check-amalgamation` job
regenerates the header and asserts byte-for-byte equality with the committed copy:

```sh
sh tools/amalgamate.sh
git diff --exit-code flow.h     # non-zero exit == drift == red build
```

This is the crown-jewel gate (a required check in branch protection). If it fails locally, you
forgot step 2. Run `make` and re-stage `flow.h`.

## Build & test

The default `make` target regenerates the header and builds the demos and examples. **It does
not run the tests.**

| Command | What it does |
| --- | --- |
| `make` | Regenerate `flow.h`, then build the 3 demos + the headless example. |
| `make test` | Regenerate, then build and run all **39** headless suites; non-zero exit on any failure. |
| `make demos` | Build `demos/hello_flow`, `demos/topo`, `demos/flowchart` only. |
| `make examples` | Build `examples/embed_headless` only. |
| `make clean` | Remove the compiled demos/examples/tests **and** `flow.h`. |

```sh
make            # regenerate + build demos + examples (NO tests)
make test       # the 39 headless suites, snapshot goldens
make demos
make examples
make clean
```

The suite list is the `TESTS=` line in the [`Makefile`](Makefile); the count there is the
authoritative number (39). Every entry maps to a `tests/test_*.c` file. The default build
flags are `-std=c11 -O2 -Wall -Wextra` with `-lm`. The headless suites cover the portable
pure-C layers тАФ model, geometry (`flow_geom`), cell buffer (`flow_cell`), render
(`flow_render`), routing (`flow_route`), layout (`flow_layout`), JSON (`flow_json`) тАФ plus
embed/term/alloc/OOM checks. The interactive run-loop (`flow_run` / `flow_feed` /
`flow_present` / `flow_term_*`) is POSIX-only and not exercised by a TTY in CI.

## Pre-commit: the sanitizer gate

CI builds and runs **every** `tests/test_*.c` under AddressSanitizer + UndefinedBehaviorSanitizer.
The default `make` build has **no** sanitizers, so run this yourself before committing. It is
the `sanitizers` job from [`.github/workflows/ci.yml`](.github/workflows/ci.yml), reproduced
verbatim тАФ run it **from the repo root** (snapshot goldens are resolved cwd-relative):

```sh
fail=0
for t in tests/test_*.c; do
  clang -std=c11 -Wall -Wextra -fsanitize=address,undefined -fno-sanitize-recover=all -g "$t" -o /tmp/sanx -lm \
    && /tmp/sanx || { echo "SANITIZER FAIL: $t"; fail=1; }
done
exit $fail
```

`-fno-sanitize-recover=all` makes the first UB/ASan hit fatal; the loop is fail-accumulating, so
it reports every failing suite rather than stopping at the first.

## C++ consumption gate

`flow.h` must remain usable from C++ тАФ both as the implementation TU and as a header linked
against a C-compiled object. The `cpp` job runs `make cpp`, which does two things:

```sh
make cpp
```

1. `tests/cpp_smoke.cpp` compiles and **runs** `flow.h` as the implementation under a C++
   compiler (`-std=c++17`) тАФ proving the enum initializers and custom-type C++ callbacks are
   well-formed in C++.
2. A C-compiled `tests/cpp_link_impl.c` object links against a C++ caller
   (`tests/cpp_link_main.cpp`) тАФ proving the `extern "C"` linkage in `src/flow_head.h` holds.

This is not part of `make test` (it needs a C++ compiler). The impl TU is compiled
`-std=c++17` without `-Wall`: the hand-tuned color-preset table in `src/flow_model.h` uses C99
array designators, which clang/gcc accept as an extension.

## Lint

Static analysis is `clang-tidy`, configured by [`.clang-tidy`](.clang-tidy) (a conservative
`bugprone-*` + `clang-analyzer-*` set). There is no `compile_commands.json`, so pass flags
after `--`, and analyze a TU that pulls in the whole library тАФ a test:

```sh
clang-tidy tests/test_smoke.c -- -std=c11
```

`HeaderFilterRegex: 'flow\.h$'` limits reports to the amalgamated library. In CI the `lint`
job is **advisory** (`continue-on-error`, `WarningsAsErrors: ''`) тАФ it informs but does not
gate.

[`.clang-format`](.clang-format) is a style **guide for brand-new files only**. Never run
`clang-format -i src/*.h`: the existing source is deliberately hand-tuned (compact one-line
aggregates, column-aligned trailing comments) and `clang-format` reflows most of it. Format a
single new file explicitly if you want, and leave existing modules alone.

## Adding a module

The library is split into dependency-ordered `src/*.h` modules. To add one:

1. Create `src/flow_yourmod.h`.
2. Append its base name to the `modules=` line in
   [`tools/amalgamate.sh`](tools/amalgamate.sh), **in dependency order** тАФ a module may only
   reference declarations from modules listed before it. The amalgamator concatenates in that
   exact order; do not reorder existing entries. The current order is:

   ```
   flow_head flow_geom flow_cell flow_model flow_undo flow_query flow_view \
   flow_layout flow_route flow_types flow_render flow_json flow_input flow_term flow_run
   ```

3. Keep the single-header contract: the non-`FLOW_IMPLEMENTATION` portion of each module is
   declarations; bodies go inside `#ifdef FLOW_IMPLEMENTATION` so exactly one TU pulls them.
   `flow_head` opens the `extern "C"` and the system-header hoist; the amalgamator closes the
   `extern "C"` at the end of `flow.h`.
4. Add a matching `tests/test_yourmod.c` suite and append its name to the `TESTS=` line in the
   [`Makefile`](Makefile). The `TESTS=` list and the on-disk `tests/test_*.c` files must stay
   in sync тАФ CI builds every file the loop finds.
5. Run `make` to regenerate `flow.h`, then `make test` and the sanitizer gate.

## Snapshot goldens

Render and model tests compare against byte-exact golden files in `tests/snapshots/*.txt`,
which are committed. [`.gitattributes`](.gitattributes) marks them `-text` so git never
normalizes line endings тАФ that would break the byte comparison cross-platform.

Any change to rendering or the model that alters output will **fail** the affected suite until
you regenerate its golden. To recapture:

```sh
rm tests/snapshots/<name>.txt     # delete the stale golden
make test                         # the suite recaptures it on the missing-file path
git diff tests/snapshots/         # REVIEW the new bytes before committing
```

Always inspect the diff тАФ a golden is only correct if the new output is the intended one.

## Conventions

These hold library-wide. The per-function detail lives in [`docs/API.md`](docs/API.md); this is
the short version a contributor needs.

### Naming

| Prefix | Meaning |
| --- | --- |
| `flow_*` | Public functions. |
| `FLOW_*` | Public constants, flags, enums, macros. |
| `flow__*` | Private тАФ implementation-only, not part of the supported surface, not documented in the API. |

### Negative-polarity gate flags

Per-node gate flags are negative-polarity: `FLOW_NODRAG`, `FLOW_NOSELECT`, `FLOW_NODELETE`
(in `src/flow_model.h`) mean *disable*. A zero-initialized node is fully permissive, so
`flow_add_node` needs no seed flags. Set them with `|=`, clear with `&= ~`; or use the public
setters (`flow_set_node_draggable` / `flow_set_node_selectable`, where `on == 0` sets the gate).
These gates are **durable** тАФ they survive undo and JSON save/load тАФ unlike the transient bits
`FLOW_SELECTED` / `FLOW_DRAGGING` / `FLOW_HOVERED`, which clear on undo.

### Pointers into the model array

`flow_get_node` (and `flow_get_edge`) return a pointer **into** the node/edge array. Any
`flow_add_node` / `flow_add_edge` may reallocate that array, invalidating the pointer:

```c
flow_node *n = flow_get_node(f, id);
flow_add_node(f, "default", pos, NULL);   /* may realloc the node array */
/* n is now DANGLING тАФ re-fetch: */
n = flow_get_node(f, id);
```

Nodes and edges are addressed by stable integer **id**, not array index. Hold ids across
mutations; re-fetch the pointer only when you need to touch fields.

### Untrusted input

`flow_load` (`src/flow_json.h`) is the only entry point that parses untrusted bytes. New JSON
parsing or any other untrusted-input handling must be hardened and covered by tests
(`test_json`, plus the OOM/alloc suites). Run the sanitizer gate on any change here.

## CI

CI is defined in [`.github/workflows/ci.yml`](.github/workflows/ci.yml). It runs on PRs and on
pushes to `main`. The gating jobs:

| Job | What it gates |
| --- | --- |
| `build-test` | Builds lib + demos + examples and runs all 39 suites across the matrix, with `-Werror`. |
| `check-amalgamation` | Regenerates `flow.h` from `src/` and asserts no drift (`git diff --exit-code flow.h`). **Required check.** |
| `sanitizers` | Builds + runs every suite under ASan + UBSan (clang, Ubuntu). |
| `cpp` | `make cpp` тАФ header compiles + runs as a C++ TU, and a C `.o` links against a C++ caller. |
| `lint` | `clang-tidy` тАФ **advisory only** (`continue-on-error`), does not gate. |

The `build-test` matrix is:

| OS | Compiler |
| --- | --- |
| `ubuntu-latest` | `gcc` |
| `ubuntu-latest` | `clang` |
| `macos-latest` | `clang` (Apple clang) |

`build-test` adds `-Werror` on top of `-Wall -Wextra`; the tree is kept warning-clean under
both gcc and clang. `-Werror` is compiler-version-fragile тАФ if a future runner's compiler
surfaces a new diagnostic, fix it in `src/` (or narrow it with a targeted `-Wno-...`) rather
than dropping the gate.

## Submitting a change тАФ checklist

```sh
# edited src/ ?
make                       # regenerate flow.h + build
make test                  # all 39 suites green

# sanitizer gate (from repo root)
fail=0; for t in tests/test_*.c; do \
  clang -std=c11 -Wall -Wextra -fsanitize=address,undefined -fno-sanitize-recover=all -g "$t" -o /tmp/sanx -lm \
    && /tmp/sanx || { echo "SANITIZER FAIL: $t"; fail=1; }; \
done; exit $fail

make cpp                   # C++ consumption gate
sh tools/amalgamate.sh && git diff --exit-code flow.h   # no drift

# stage src/ + the regenerated flow.h together; never `git add .`
git add src/flow_*.h flow.h tests/test_*.c
```

For terminal-rendering caveats (box-drawing edges, one glyph = one cell, codepoint-per-cell
label measurement) see the "Honest terminal limits" note in [`README.md`](README.md). For the
full feature list and per-function reference, see [`README.md`](README.md) and
[`docs/API.md`](docs/API.md).
