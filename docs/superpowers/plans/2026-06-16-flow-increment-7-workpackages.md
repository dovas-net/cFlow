# flow — Increment-7 work packages: the feel layer

Five work packages + a final integration pass turning `flow` from a functional node-graph engine
into one that **feels** finished: a `flow_theme` color-token model with light/dark presets (the
DEFAULT preset reproduces today's hardcoded `FLOW_FG/FLOW_BG`/grid literals **byte-for-byte**, so
every existing golden survives), live **connection feedback** (the rubber-band and the candidate
handle recolor green/red the instant a drop would be accepted/rejected), and the engine's first
**interactive built-in widgets** — a Controls bar (`[+][-][fit][lock]`) and node/edge toolbars —
riding a single new **widget hit-test seam** at the top of the mouse-input classification path
(a click on an engine-owned widget cell routes to a widget handler and does NOT fall through to
the canvas). The minimap is render-only today; this increment makes built-in chrome clickable.

Each package section was drafted against the LANDED code on `increment-6` @ `3a33ca2` by a verified
recon fan-out (struct/flags/keys, theme call-sites, input-path/zoom, connection accept-path,
anchor/build discipline) grounded in source with file:line citations; re-verify line numbers at
implementation time (earlier packages shift later lines — theme-foundation rewrites every color
call-site in `flow_render.h`/`flow_types.h`, so the seam packages that follow it must rebase).

**Approved going in** (do not re-litigate; the Open-questions section surfaces only genuine
sub-forks): inc-7 is the FEEL layer = five packages in the order below; the widget hit-test seam
is introduced by controls-bar and reused by both toolbars; the theme model is presets + a fixed
token set ONLY (no per-element override table); lock is a flow-level bool (controls' 4th button);
connection snap-radius is DEFERRED; no node-flag bit is consumed (theme/lock/widgets/conn_valid
are all `struct flow` state); theme/color_mode/lock/widget-config/conn_valid are TRANSIENT chrome

### 1. Theme foundation — color tokens + light/dark presets  `[M]`  ·  id: `theme-foundation`

**Goal.** The engine has no theme: every render call-site hard-codes the three raw color literals `FLOW_FG` (=7), `FLOW_BG` (=0), and `FLOW_BG_GRID_FG` (=8). `FLOW_FG`/`FLOW_BG` are `#define`s in `src/flow_cell.h:3-4`; `FLOW_BG_GRID_FG` is a render-local `#define` in `src/flow_render.h:8`. There is no run-time color state on `struct flow` (grep for `theme|color_mode` across `src/*.h` returns nothing) and no way for an app to pick a dark vs light palette without recompiling. The fix introduces a `flow_theme` token struct as a value member on `struct flow` (beside the `minimap`/`bg` config sub-structs at `src/flow_model.h:363-364`), seeded in `flow_new` to a DEFAULT preset whose token values **equal the current literals verbatim** (fg 7 / bg 0 / grid 8), and converts every ENGINE-CHROME render call-site that takes a `flow_t *f` (all in `src/flow_render.h`) from the literal `#define`s to `f->theme.*` lookups. A new `flow_set_color_mode(f, FLOW_COLOR_LIGHT|FLOW_COLOR_DARK)` re-seeds the token struct from a fixed preset table. The token set is fixed and minimal — no per-element override table (deferred, YAGNI). Two of the tokens (`handle_valid`, `handle_invalid`) are DEFINED and DEFAULTED here but CONSUMED by package 2 (`connect-feedback`); the widget-chrome tokens are defined for packages 3-5.

**User value.** An app author calls `flow_set_color_mode(f, FLOW_COLOR_DARK)` once and the engine-drawn chrome — canvas fill, background grid, edges, handle markers, marquee, alignment guides, minimap panel, statusbar — re-colors as a coherent palette, instead of being frozen at the single hardcoded scheme. The default is unchanged for every existing embedder (byte-identical output until they opt in), so the feature is invisible until armed. **Design notes.**

*The byte-identical seed is an EXPLICIT non-zero default — theme is the INVERSE of the inc-5 "calloc OFF = byte-identical" precedent, and conflating them is the trap.* The frame name-checks inc-5's discipline where a new opt-in feature defaults OFF via calloc-zero and is therefore byte-identical (helper-lines: `int helper_on; /* calloc OFF */`, `src/flow_model.h:352`). Theme does NOT work that way and the section must say so plainly: a calloc-zeroed `flow_theme` is `{fg:0, bg:0, grid_fg:0, ...}`, which would paint every `FLOW_FG`-colored cell with fg index 0 (black-on-black) and break EVERY existing golden the instant the conversion lands. Byte-identity holds for the OPPOSITE reason — `flow_new` EXPLICITLY seeds the DEFAULT preset to the legacy literal values, exactly the pattern `flow_new` already uses for the other field whose correct default is non-zero (`f->tick_ms = 100;`, `src/flow_model.h:556`; the whole `551-557` block exists precisely for non-zero defaults). The default preset is, verbatim: `fg = FLOW_FG (7)`, `bg = FLOW_BG (0)`, `grid_fg = FLOW_BG_GRID_FG (8)`, `handle = FLOW_FG`, `edge_fg = FLOW_FG`, `widget_fg = FLOW_FG`, `widget_bg = FLOW_BG`, and the pkg2/accent tokens to sensible values that are not yet read. Because every converted call-site reads `f->theme.fg`/`.bg`/`.grid_fg` and those equal `7`/`0`/`8` after the seed, `flow_render`'s output is bit-for-bit what it was — the conversion is a pure indirection with identical values. This is the discipline the frame pins: the default preset reproduces today's hardcoded literals EXACTLY so every existing golden stays byte-identical.

*The f-reachability split is the whole shape of the conversion (and the scope line, not a fork).* The literals appear in TWO classes of call-site, and only one class can reach `f->theme`. CLASS A — engine chrome — lives entirely in `src/flow_render.h`, inside functions that take `flow_t *f` (`flow_render`, `flow__background`, `flow__minimap`); every literal there is f-reachable and IS converted. CLASS B — node-type `render()` bodies — lives in `src/flow_types.h` (`flow__default_render` at `17`/`20`/`21`, `flow__group_render` at `42`/`45`/`46`), and these are vtable callbacks with signature `void (*render)(const flow_node*, flow_surface*, flow_render_ctx)` (`src/flow_model.h:24`), where `flow_render_ctx` is `{ float zoom; unsigned flags; int lod; }` (`src/flow_model.h:21`) — NO `flow_t *` and NO theme handle. A built-in node renderer CANNOT read `f->theme` without an ABI change to a contract every app compiles against. So CLASS B is deliberately left on the `FLOW_FG`/`FLOW_BG` `#define`s, which stay PUBLIC and equal to the default preset. The scope line is principled, not an apology: **the theme governs ENGINE-OWNED CHROME; node-type renderers — built-in and app alike — own their own color via the public `FLOW_FG`/`FLOW_BG` constants.** This is exactly why the recon flags the demos: `demos/topo.c:43-159` uses `FLOW_FG`/`FLOW_BG` in its custom node `render()` body (`dev_render`) AND its detail-popup overlay, while `demos/hello_flow.c:61-84` and `demos/flowchart.c:191-219` use the default node type and touch the literals only in their `on_overlay` HUDs — all of these stay valid and unchanged, because the constants remain public defaults. The frame's test is satisfied verbatim by CLASS A alone: a swap recolors the canvas clear, a grid cell, an edge cell, a handle cell, and the statusbar row; node bodies don't move, and the token set doesn't name a "node body" color.

*Why NOT thread theme through `flow_render_ctx` (the rejected ABI-append).* The tempting "complete" fix adds a `const flow_theme *theme;` field to `flow_render_ctx` (`src/flow_model.h:21`) so node bodies can read it. Rejected for three reasons. FIRST, it mutates the node-render contract every app compiles against — `flow_render_ctx` is passed by value to `nt->render` at `src/flow_render.h:238`, and growing it is an ABI change for a node-body recolor the fixed token set does NOT name and the frame's test does NOT require. SECOND, it is the precise conservatism inc-6 #4 chose when it declined to append `on_tick` to the `flow_callbacks` tail ("the callbacks tail is untouched … no ABI-append this increment") — same call, same increment-discipline. THIRD, node-body color is genuinely app-owned: an app's `render()` already passes whatever color it wants to `flow_box`/`flow_text`; forcing it through engine theme tokens would be a policy the engine has no business imposing. If a future increment wants themed node bodies, the clean route is a NEW ctx field at that time with its own justification — not smuggled in here.

*modules= fold — no new module; tokens land in flow_cell.h, and `FLOW_BG_GRID_FG` moves UP a module.* The recon recommends folding theme tokens into `src/flow_cell.h` rather than adding a module, and that is correct: `amalgamate.sh:5` lists `flow_head flow_geom flow_cell flow_model ...` — `flow_cell` is module 3, ahead of `flow_model` (4), `flow_render` (11), and `flow_types` (10), all of which consume color tokens. Putting `flow_theme`/`flow_color_mode` in flow_cell.h makes them visible to every downstream module with ZERO `amalgamate.sh` edit and no forward-declaration hazard. The one real relocation: `FLOW_BG_GRID_FG` currently first appears as a `#define` in `flow_render.h:8` (module 11), but `flow_new` (in `flow_model.h`, module 4) must seed `theme.grid_fg = FLOW_BG_GRID_FG` — module 4 cannot see a macro that first appears in module 11. So `FLOW_BG_GRID_FG` moves UP to `flow_cell.h` (beside `FLOW_FG`/`FLOW_BG`), same value `8`, and its explanatory comment (`src/flow_render.h:5-7`, "a dim 256-color fg, NOT FLOW_DIM") moves with it. `FLOW_DASH_PERIOD` (`src/flow_render.h:13`) is an animation cadence, NOT a color, and STAYS render-local — do not touch it. A NEW module would force an `amalgamate.sh:5` edit and careful re-ordering; the fold avoids both. Placement splits cleanly along what-needs-whom: the TYPE defs (`flow_theme`, `flow_color_mode`) and the relocated `FLOW_BG_GRID_FG` go in **flow_cell.h** (module 3) because `struct flow`'s `flow_theme theme;` member and `flow_new`'s seed both live in module 4 and need the type + the grid macro already visible. The public SETTER DECLS (`flow_set_color_mode`/`flow_color_mode_get`) go in **flow_model.h**'s config cluster beside `flow_set_background` (`src/flow_model.h:313-315`) — the house location for every `flow_set_*` declaration, which also sidesteps any question of whether `flow_t` is forward-declared by the time module 3 is concatenated. The setter BODY lands in flow_model.h beside the other `flow_set_*` bodies (`src/flow_model.h:998-1003`), keeping decl and body co-located.

*The preset table and the setter — a value-copy re-seed, no heap; and colorMode's whole point is the BACKGROUND, not the foreground.* `flow_set_color_mode` is a switch over the three modes that writes a whole `flow_theme` literal into `f->theme` (and records `f->color_mode = mode`), mirroring `flow_set_background`'s direct-write body (`src/flow_model.h:1001-1003`). The preset DIRECTIONS are load-bearing, not a free tuning detail — get them backwards and the feature is a near-no-op. xyflow's `colorMode` exists to flip the canvas: LIGHT = light background / dark foreground, DARK = dark background / light foreground. Today's hardcoded scheme is `bg 0` (black) with `fg 7` — already a DARK aesthetic — so the mapping is: **DEFAULT** = legacy `fg 7 / bg 0 / grid 8` (byte-identical, non-negotiable); **DARK** = the dark-background palette nearest legacy (e.g. `fg 15 / bg 0 / grid 8` — brighter fg on the same dark canvas); **LIGHT** = the background FLIPS to a light index and fg goes dark (e.g. `bg 15 / fg 0 / grid 7`). The discriminating axis is `bg`: a preset that leaves `bg` at 0 is indistinguishable from DEFAULT in the one dimension colorMode is for, so the cell-field test (block 2) asserts the `bg` flip, not just an `fg` nudge. The "no surprise to existing apps" guarantee is satisfied by DEFAULT alone — an app that never calls `flow_set_color_mode` stays at the byte-identical legacy palette; an app that calls `flow_set_color_mode(FLOW_COLOR_LIGHT)` is opting in and WANTS the flip. The only hard contract is that `FLOW_COLOR_DEFAULT` equals the `flow_new` seed (round-trip byte-identical); the exact non-default index choices are tunable, but their DIRECTION (LIGHT flips bg light, DARK keeps bg dark) is not. `flow_theme` is a pure value type — `uint8_t` scalars, no pointers — so it inherits zero-cost teardown via the single `free(f)` in `flow_free` (`src/flow_model.h:571-577`); NO entry is added to `flow_free`, matching the minimap/bg value-member precedent (the only heap owners stay nodes/edges/clip/journal).

*Transient view-state: never saved, never journaled (the approved-design pin).* `theme` and `color_mode` are chrome/view state, exactly like the inc-6 tick/modal precedent (transient, calloc-or-seed, no journal, no save). `flow_save` (`src/flow_json.h`) is untouched — it never emitted color and must not start; `color_mode` is re-seeded by the app after `flow_load` if it wants non-default (the `FLOW_HIDDEN`/`FLOW_ANIMATED` re-arm convention). No `flow_callbacks` field, no undo op, no JSON key. This keeps `flow_save`'s goldens and the JSON round-trip tests byte-identical.

*Defined-but-unwired tokens, and the selection-accent trap.* Six tokens are added now but read later: `handle_valid`/`handle_invalid` (pkg2 connect-feedback recolors the candidate ◉ at `src/flow_render.h:254`), `widget_fg`/`widget_bg` (pkg3-5 controls/toolbars chrome). All FOUR are consumed at f-reachable sites when wired (the widgets and the candidate-handle recolor all live in functions taking `flow_t *f`), so there is no ctx issue downstream — they are defined here purely so the token set is fixed in one place. The `accent` token is the trap: selection emphasis today is a FLOW_BOLD ATTRIBUTE (`src/flow_render.h:194` edge, `src/flow_types.h:15`/`40` node), NOT a color. `accent` is DEFINED for completeness but LEFT UNWIRED this increment — wiring a selection COLOR would change `.fg`/`.bg` on every cell of every selected node/edge and MOVE every golden that contains a selection (e.g. the select/marquee/groups goldens). Selection stays attribute-based; the token waits for a future increment that explicitly accepts that golden churn.

### 2. Live connection-validity feedback (candidate handle recolors valid/invalid)  `[S]`  ·  id: `connect-feedback`

**Goal.** The headline FEEL gap of the increment, verified by a real run: during an in-flight connection (`f->conn_active`), the candidate target handle ◉ is rendered **byte-identical** whether the drop would be accepted or rejected — a duplicate-edge target and a fresh valid target both draw `0x25C9` in `FLOW_FG`/`FLOW_BG` (`src/flow_render.h:254`). The engine already *knows* the answer twice over — the structural rejects + validator gate in `flow_add_edge` (`src/flow_model.h:795-803`) — but that knowledge is computed only at COMMIT time and is never surfaced while the cursor hovers. The live track path discards even the candidate handle: `flow_update_connection` calls `flow_hit_handle(f, screen, &tn)` but throws the returned handle index away, keeping only the candidate node `tn` (`src/flow_model.h:1554`). Fix: **factor a single accept predicate** `flow__connection_would_accept` out of `flow_add_edge:795-803` (so preview and commit can never drift), call it from the hover path on the *resolved* target handle each frame, cache the result (`conn_valid`) plus the candidate node+handle on `struct flow` beside the existing `conn_*` cluster (`src/flow_model.h:357`), and recolor the candidate ◉ at `src/flow_render.h:254` in the new `theme.handle_valid` / `theme.handle_invalid` tokens (package 1). Add a read-only accessor `flow_connection_valid(f)`.

**User value.** The end-user gets the standard xyflow connection affordance: as you drag a wire, the handle under the cursor turns green when releasing there would make an edge and red when it would not (self-edge, duplicate, or an app `isValidConnection` rejection) — feedback *before* the drop, not a silent no-op after it. The app author gets it for free: any graph with handles shows the cue with zero new wiring, and a headless app/test can read the same truth via `flow_connection_valid(f)` after `flow_update_connection`. Because the cue is a foreground-color swap on one cell, every node renderer, demo HUD, and existing golden is untouched (see Test plan — the snapshot encoding is glyph-only).

**API additions.**
```c

   when the current candidate drop (resolved target handle under the cursor) WOULD be
   accepted by flow__connection_would_accept — i.e. flow_end_connection on this target
   right now would create an edge. 0 when no connection is active, or the drop would be
   rejected (self-edge, duplicate, or validator veto). Recomputed each motion frame by
   flow_update_connection; this is a thin read of f->conn_valid. */
int flow_connection_valid(flow_t *f);

/* struct flow — appended to the in-flight connection cluster at src/flow_model.h:357,
   beside `int conn_active, conn_node; char conn_handle[16]; flow_pt conn_end;`.
   All TRANSIENT: never saved (flow_save unaffected), never journaled (conn_* precedent).

/* Private (flow__ prefix), both factored from existing inlined logic — declared and
   defined in flow_model.h, no module/amalgamate change:
     the SINGLE accept predicate, mirroring flow_add_edge:795-803 (handles normalized
     to "" internally like :797). exclude_edge = an edge id to skip in the duplicate
     scan (-1 for a fresh connect; the reconnected edge's id for flow_reconnect_edge),
     so the ONE predicate serves add, preview, AND reconnect. */
static int  flow__connection_would_accept(flow_t *f, int src, int dst,

/* the target-handle defaulting from flow_end_connection:1564-1570: explicit `handle`
```

**Design notes.**

*The handle-defaulting divergence is the #1 correctness trap — and a lone predicate WOULD reproduce the very bug this package kills.* `flow_add_edge` takes ALREADY-RESOLVED handles; `flow_end_connection` resolves the target handle (`:1564-1570`) BEFORE calling `flow_add_edge` (`:1572`). If the hover-check fed the predicate the raw cursor handle (empty when the cursor is over a node BODY rather than a specific handle), then a body-hover over a node already connected by `A→B(out,in)` would pass `th = ""`; the duplicate scan (`:798-800` compares `target_handle` to the normalized `ths`) finds no match against the stored `"in"`, the predicate returns ACCEPT, and the preview shows **green while the actual commit defaults `th` to `"in"` and rejects as a duplicate** — the exact preview/commit drift this package exists to eliminate. The fix is to share TWO statics, not one: `flow__resolve_target_handle` (the defaulting, lifted from `:1564-1570`) and `flow__connection_would_accept` (the gate, lifted from `:795-803`). Both `flow_end_connection` and `flow_update_connection` do resolve-THEN-check on the identical resolved handle, so the preview is computed from byte-identical inputs to the commit. This is *why* a shared predicate — drift between preview and commit is not a style concern, it is a wrong-color bug, and the defaulting is the subtle half that an "extract the rejects" reading misses.

*Why `flow_add_edge` must CALL the predicate, not run a parallel copy.* The anti-drift guarantee only holds if there is ONE source of truth. So `flow_add_edge:795-803` is replaced by `if (!flow__connection_would_accept(f, src, dst, shs, ths, -1)) return -1;` (the handle normalization at `:797` moves INTO the predicate so both callers normalize identically; `flow_add_edge` keeps passing `shs`/`ths` which are then re-normalized idempotently, or passes `sh`/`th` raw and lets the predicate normalize — either is fine, the predicate is the only normalizer). A parallel predicate that merely *mirrors* the rejects can silently diverge the first time someone edits one copy. The behavior-preserving proof is free: the existing `test_connect.c` self/duplicate/validator block (`tests/test_connect.c:199-220`) must stay green across the refactor — if the extraction changed any accept/reject outcome, that block fails. `flow_reconnect_edge` (`:1020-1029`) is the second copy; the `exclude_edge` param lets it unify too (its duplicate scan skips the edge being reconnected, `:1022`). Recommendation: wire all three to the one predicate (one true source). Scope call: if `[S]` budget is tight, the reconnect unification is the cuttable leg — leave the `exclude_edge` param in the signature and wire `flow_add_edge` (`-1`) + the hover-check now; reconnect's wire-up is then a one-line later change, and the param means no signature churn when it lands.

*`conn_valid` is necessary but NOT sufficient for render — the candidate ◉ must be IDENTIFIABLE.* A bool tells you valid/invalid; it does not tell the render loop WHICH ◉ to recolor. `flow_update_connection` today keeps only the candidate NODE (`tn`) and discards the handle index from `flow_hit_handle` (`:1554`) — so the candidate target handle is not reachable from existing state. The candidate handle is exactly the RESOLVED target handle (`flow__resolve_target_handle` of whatever the cursor is over, hit-handle id or defaulted), the same value that feeds the validity check. **The resolve input is the un-discarded hit-handle index** — `flow_update_connection` must capture `int hidx = flow_hit_handle(f, screen, &tn)` (today thrown away at `:1554`) and pass the SPECIFIC handle id when the cursor is over a handle (`hidx >= 0 ? flow_node_handle_at(f, tn, hidx)->id : NULL`), letting `flow__resolve_target_handle` default ONLY on a body hover. This is a latent-bug trap on multi-target-handle nodes: a node with two target handles `in1`/`in2` already connected by `A→B(out,in1)` should color VALID when the cursor is specifically over `in2` — but an implementation that ignores `hidx` and always defaults resolves to `in1`, calls it a duplicate, and colors invalid. On the single-target-handle nodes every demo/test uses, hit-and-default collapse to the same id, so an always-default implementation passes every single-handle assertion while shipping this bug — exactly why the index must NOT be discarded and why block 2 includes a two-target-handle node. So the cache is three fields beside the `conn_*` cluster (`:357`): `conn_valid`, `conn_target_node`, `conn_target_handle[16]`, all written once per motion frame in `flow_update_connection`. The marker loop (`:245-256`) then recolors the cell when `n->id == f->conn_target_node && strncmp(h->id, f->conn_target_handle, sizeof h->id) == 0` — the identical match shape the source-bold check already uses against `f->conn_handle` at `:252-253`. The alternative — re-run `flow_hit_handle` + the predicate INSIDE `flow_render` — is rejected: it computes the candidate twice per frame and puts a hit-test (a model query) into the render path, breaking the "render is a pure function of (model, view, tick)" determinism contract the inc-6 frame pins. Cache it in the input/update layer where the cursor already lives; render only reads.

*Compute `conn_valid` for ALL cases before `flow_update_connection`'s existing return — including the negatives.* The function already classifies: `tn` is a handle-hit node, else a body-hit node, else `-1`, and it returns `-1` when `tn == f->conn_node` (`:1557`). The cache mirrors that: when there is no candidate (`tn == -1` or `tn == f->conn_node`), set `conn_target_node = -1`, `conn_target_handle[0] = 0`, `conn_valid = 0` (no candidate ⇒ nothing to accept). When there IS a candidate node, resolve its target handle and set `conn_valid = flow__connection_would_accept(f, f->conn_node, tn, f->conn_handle, resolved_th, -1)`. The self-edge case (`tn == f->conn_node`) never reaches the predicate because `flow_update_connection` already filters it — but if a future change let it through, the predicate's own self-reject (`:795`) returns 0, so the cue stays correct by construction. The candidate-node `FLOW_HOVERED` set at `:1556` is unchanged (it is what reveals the candidate's handles so the ◉ is even drawn and hittable — the shared `flow__node_handles_visible` gate at `src/flow_model.h:1455-1457`).

*Render: a fg-only recolor of one cell, handle-only for v1 (rubber-band tint deferred).* The candidate ◉ put at `:254` currently passes `(FLOW_FG, FLOW_BG, bold)`. The recolor selects the fg per validity: `fg = is_candidate ? (f->conn_valid ? f->theme.handle_valid : f->theme.handle_invalid) : FLOW_FG` (the `FLOW_FG` literal becomes `f->theme.handle_*`/default per package 1's call-site conversion — see Depends on). The source ◉ (distinguished by `FLOW_BOLD` at `:252-253`) is NOT a candidate and keeps its color; a node can be neither/both, and the match conditions are disjoint (`conn_node` vs `conn_target_node`). I recommend **handle-only feedback for v1** and explicitly DEFER tinting the dashed rubber-band (`:281-297`, the `flow_cellbuf_put` at `:294`): (a) the rubber-band is the in-flight wire's whole length and tinting it red on every body-hover would be visually noisy and would fight the grid backdrop the dashes show through; (b) the handle is where the drop lands, so it is the precise locus of the affordance; (c) it keeps the change to a single cell and a single literal-fg swap, preserving the `[S]` size. The rubber-band tint is a clean follow-up (same `conn_valid` read, same token) recorded here, not built.

*LOD-1 suppression — recommend suppress, parity with the increment's animation discipline.* At LOD 1 nodes collapse to a 1×1 marker (`flow__node_footprint` at the collapsed size) and `flow__handle_screen` clamps every handle of a node to that single cell, so the candidate ◉ is neither spatially distinct nor legible as "the handle under the cursor." Recolor there would tint the collapsed marker, reading as a state change on the node, not a connection cue. Recommendation: gate the recolor on `flow__lod_for(f, f->view.zoom) == 0` (the same LOD-0 gate inc-6 #5 uses to suppress marching ants, `src/flow_render.h`), drawing the plain `FLOW_FG` ◉ at LOD 1. This is an OPEN QUESTION below (the alt — recolor at all LODs — is defensible if one argues "any cue beats none," but it fights legibility at overview zoom). The validity COMPUTATION (`flow_connection_valid`) is LOD-independent — only the visual cue is suppressed — so headless/test truth is unaffected.

*Forward-declaration ordering (the amalgamation hazard).* `flow_add_edge` (`:794`) must CALL `flow__connection_would_accept`, but the predicate is lifted from logic that, together with its other callers (`flow_reconnect_edge` `:1012`, `flow_end_connection` `:1559`, `flow_update_connection` `:1551`), sits AFTER `flow_add_edge`. So both statics need a forward declaration placed before `:794` (or define both `flow__connection_would_accept` and `flow__resolve_target_handle` in full immediately above `flow_add_edge`). Same with the amalgamated single-header: `flow_model` is one translation unit, so a use-before-definition is a hard compile error — declare-before-use is mandatory, not stylistic.

*Flag ledger + transient/JSON pins (stated outright).* NO node/edge flag bit is consumed: `conn_valid`, `conn_target_node`, `conn_target_handle` are flow-level state on `struct flow`, exactly like the existing `conn_*` cluster — bit 64u stays reserved for the deferred per-element interaction gates (the FLAG LEDGER pin). All three are TRANSIENT: never emitted by `flow_save` (it does not even emit in-flight connection state) and never journaled (the `conn_*`/marquee/helper precedent — connection state is gesture-local). `conn_target_node` needs an explicit `-1` init in `flow_new` beside the existing `f->conn_node = -1` (`src/flow_model.h:553`); `conn_valid` and `conn_target_handle` are correct at calloc-zero (off / empty) and need no `flow_new` line. None is heap-owning — they inherit the zero-cost teardown of the single `free(f)`.

### 3. Controls bar + the interactive-widget seam (lock mode)  `[L]`  ·  id: `controls-bar`

**Goal.** The engine has ZERO interactive built-in chrome: the minimap is render-only (`flow__minimap`, `src/flow_render.h:33-81`, never hit-tested), and every mouse press falls straight into canvas classification (`flow_handle_mouse`, `src/flow_input.h:74`). xyflow ships a corner `<Controls>` panel — `[+][-][fit-view][lock]` — that an end-user clicks; flow has no equivalent and no path for one, because the press handler has no branch ABOVE the graph-element trio (`src/flow_input.h:124-126`) where screen-space chrome must intercept. This package lands TWO coupled things. **(a) The interactive-widget seam:** a hit-test branch at the TOP of the left-press block (`src/flow_input.h:105`, immediately after `f->mouse_down` is armed at `:123`, BEFORE the conn-resolve at `:114`, space-pan at `:117`, and the handle/edge/node trio at `:127+`) that compares the press cell `scr` (`src/flow_input.h:90`, already 0-based screen cells) against a small set of engine-owned **screen-space** widget rects; a hit routes to a widget handler, performs the press-consume reset (mirroring the handle-grab at `src/flow_input.h:134-136`), and RETURNS — the press never falls through to the canvas. **(b) The Controls widget itself:** a corner-anchored `[+][-][fit][lock]` row enabled by `flow_set_controls(f, enabled, corner)` (reusing `flow_corner`, `src/flow_model.h:310`), drawn in the compose order AFTER the minimap (`src/flow_render.h:336`) and BEFORE the app overlay (`:337`), whose render pass POPULATES the widget rect cache the press path reads. The `+`/`-` cells call `flow_zoom_in`/`flow_zoom_out` (`src/flow_view.h:30-31`) centered on `{f->cols/2, f->rows/2}` (the keyboard-zoom convention, `src/flow_run.h:67-68`), `fit` calls `flow_fit_view(f, 2)` (`src/flow_model.h:1272`, the margin the `f` built-in uses, `src/flow_model.h:1396`), and `lock` toggles a new flow-level `int locked;` (`src/flow_model.h:357` cluster). When `locked` is set, the engine suppresses node-drag arming, connect arming, reconnect arming, marquee, and click-selection changes via a single locked-press handler at the top of the left-press block (`src/flow_input.h:105`) plus a release-click guard (`:368`), while pan/zoom keep working — the xyflow Controls-lock semantics. This is the engine's FIRST interactive built-in; the seam it introduces is the one the node-toolbar (#4) and edge-toolbar (#5) reuse.

**User value.** An app author gets a drop-in viewport control panel with one call — `flow_set_controls(f, 1, FLOW_CORNER_BL)` — and the end-user can zoom, fit, and lock the canvas with the mouse, no keyboard required (the headline xyflow Controls affordance, previously impossible because no engine widget could receive a click). Lock mode is the "presentation / read-only" gesture: a viewer can pan and zoom around a graph without any risk of accidentally dragging a node, re-selecting, or starting a stray connection — exactly the protection a shared or kiosk view needs. And every later package that wants clickable chrome (the two toolbars) inherits a proven, single-seam classification path instead of re-deriving "how does a click reach a button," so the cost of interactivity is paid once here.

**Design notes.**

*The seam contract: WHY the hit-test sits above canvas classification, and WHY it consumes with no fall-through (the load-bearing decision).* The widget is **screen-space chrome** — it occludes whatever graph content is under it and must win the press unconditionally. The recon's crux (`input-path-zoom`) is the screen-vs-world distinction: the trio comment at `src/flow_input.h:124-126` documents the insertion point for a WORLD-space graph-element test (the edge package legitimately inserted between handle and node-body there, `src/flow_input.h:139-158`), because edge endpoints are graph geometry that belongs in the handle→node→pane precedence chain. A Controls button is NOT graph geometry; placing its test in the trio would let a press that lands on the bar be stolen by a mode handler that runs FIRST: the conn-resolve at `src/flow_input.h:114` (a mid-connection click on the bar would complete/cancel the connection instead of zooming), or the space-pan arm at `src/flow_input.h:117` (a space-held click on the bar would arm a pan). So the test sits at the TOP of the `button==0` block. **The consume rule** is the second half of the contract: a hit calls the handler and returns immediately after the `134-136` reset, so the press neither arms a drag/marquee/pan nor reaches the release click branch (`src/flow_input.h:349`, which only fires on `mouse_down && !moved` — cleared by the reset). This is the same "consume, do not fall through" property the handle-grab already proves (`src/flow_input.h:130-137`); the widget reuses its exact reset so the no-op-release behavior is inherited, not re-invented.

*The exact ordering subtlety — arm `mouse_down` first, then hit-test, vs hit-test before arming.* The block opens by NOT arming until `:123` (`f->mouse_down = 1; …`); the conn-resolve at `:114` runs BEFORE that arm. I place the widget hit-test at the very top of the block, BEFORE the conn-resolve, and have the handler perform the FULL `134-136` reset (which sets `mouse_down = 0` etc.) rather than relying on the later arm. Rationale: a click on a Controls button while a connectOnClick is in flight must NOT resolve the connection — the user is reaching for the zoom button, not completing a wire. Putting the test first and resetting in the handler makes the bar win even mid-connection, and the reset leaves connection state for the lone-ESC/elsewhere to cancel (the widget does not touch `conn_active` — it is orthogonal chrome). Trap recorded: do NOT place the test after `:123`'s arm only, because the conn-resolve at `:114` would already have consumed a mid-connection click before the test ran. The test is the first thing in the `button==0` body.

*Right-click and the 90-91 alternative (rejected — left-click only).* The recon flags `src/flow_input.h:90-91` (press-body entry, before the `button==2` context dispatch at `:92-104`) as the alternative insertion point IF the widget must also pre-empt right-click. It does not. The Controls bar is left-click-only chrome (xyflow's buttons are left-click); a right-click on the bar should fall through to the normal context path (which finds no node/edge under the bar's screen cells and harmlessly fires nothing, `src/flow_input.h:96-103`). Pre-empting right-click would mean intercepting `button==2` for no behavior, adding a second hit-test call site. So the test lives inside the `button==0` block (`:105`), NOT at `:90`. Stated explicitly because the toolbars (#4/#5) may later want right-click on a toolbar button — that is their decision to make at the same seam, not pre-committed here.

*Widget rect storage: a render-filled cache, NOT recomputed in the input path (the chosen shape, justified against the recompute alternative).* The press hit-test needs the buttons' SCREEN rects, but the press handler runs at a DIFFERENT time than render and does not know the corner offset (which depends on `cols`/`rows`, exactly as `flow__minimap` computes `ox/oy` from `cb->w/cb->h` at `src/flow_render.h:37-42`). Two shapes were available: **(A)** recompute the rects in the input path each press (re-derive the corner anchor from `f->cols`/`f->rows` + the fixed glyph layout), mirroring how anchors are recomputed each render; **(B)** a tiny fixed cache on `struct flow`, filled by the render pass, read by the press. I choose **(B)**, and the reason is single-source-of-geometry: the render pass is where the bar's cells are actually drawn, so if render and hit-test compute the rects independently (shape A) they can DRIFT — the same drift the engine deliberately avoids by sharing `flow__handle_screen` between the handle render loop and `flow_hit_handle` (`src/flow_model.h:1446-1453`, the "they MUST agree" invariant). With (B), the render pass emits the cells AND records each cell-run as a `widgets[]` rect in the same loop, so the hittable region is BY CONSTRUCTION the drawn region. The cache is a FIXED value array (no heap, the `keys[32]` precedent at `src/flow_model.h:365`), so it inherits zero-cost teardown via the single `free(f)` in `flow_free` (`src/flow_model.h:576`) and needs no `flow_free` edit. It is refilled every render (set `nwidgets = 0` at the top of the pass, like `f->helper.nvert = 0` before a guide sweep, `src/flow_input.h:270`), so a stale rect cannot survive a frame where controls are disabled. The cost — a per-frame fill of ≤4 rects — is negligible, and crucially this is the shape the TOOLBARS extend: #4/#5 push their (selection-derived, per-frame) button rects into the SAME `widgets[]` list, so the seam's hit-test scans one array and routes by `action`, and the toolbars never touch the input path at all. (If a frame is not presented — e.g. a headless test that calls `flow_handle_mouse` without `flow_render` — the cache is empty and the press falls through; tests render first, see Test plan.)

*Lock gating — the semantic target and the chosen TWO-POINT realization (decision stated, not derived).* The approved design pins lock = "suppress node-drag + connect-arming + click-selection changes, while pan/zoom still work." The `button==0` block (`src/flow_input.h:105-173`) has FIVE press-arm paths that each lead to a graph mutation or a selection change, and a complete gate must cover all five: handle→connect-begin (`:127-138`), edge-endpoint→reconnect (`:143-158`, which also opens an undo bracket and returns before any node test), node-body→drag/modifier-select (`:159-168`), shift-empty→marquee (`:170-171`), and the no-move release click→select/edge-select/pane-click (`:349-386`). Three scattered `!f->locked` gates (at connect `:130`, node-candidate `:159`, release `:368`) MISS two of them: the reconnect arm sits between `:130` and `:159` and slips through, and once `down_node` is gated to `-1` a shift-press now ALWAYS falls to the marquee arm at `:170` — so a "three-point gate" would still let a locked canvas repoint edges and marquee-select. **Recommended (the chosen shape): a single locked-press handler at the TOP of the `button==0` block, after the widget hit-test, before the conn-resolve at `:114`.** It ARMS tracking exactly like the normal arm at `:123` (`f->mouse_down = 1; f->moved = 0; f->down_pos = scr;`) but ZEROES all classification (`f->down_node = -1; f->drag_node = -1; f->dragging_pan = 0; f->down_modsel = 0; f->marquee_active = 0;`) and `return`s. Returning there preempts handle, reconnect, node, modifier-select, AND marquee in ONE place — every mutation/selection arm is upstream-blocked. **Pan survives** because tracking stays armed: the motion handler bails only on `if (!f->mouse_down) return;` (`src/flow_input.h:192`), and since `mouse_down` is 1 a drag with `down_node == -1` and `marquee_active == 0` routes to the `else { f->dragging_pan = 1; … }` branch (`:208-210`) — so a locked drag-anywhere PANS (a deliberate, documentable consequence: lock is a graph/selection lock, not a view lock; the pane-pan-anywhere behavior is the intended "move the camera freely" affordance). Zoom is likewise ungated (the +/- keys at `src/flow_run.h:67-68` and the `[+]/[-]` widgets bypass this block entirely). The ONE remaining leak the tracking-arm does NOT close is the no-move CLICK: a locked press-then-release with no motion reaches the release click branch (`src/flow_input.h:349`, fires on `mouse_down && !moved`) and would hit the `else` edge/pane branch at `:368` (`flow_hit_edge` → edge-select or `flow_clear_selection` + `on_pane_click`). So the second (and final) edit guards that branch: wrap `:368`'s `else` as `else if (!f->locked)` so a locked click neither edge-selects nor clears selection nor fires `on_pane_click`. **Two edit points, complete:** the locked-press tracking-arm at the top of `:105` (preempts all five mutation/selection arms, preserves pan) + the `:368` release guard (closes the no-move click). The lock handler runs AFTER the widget hit-test so the `[lock]` button itself is always clickable to UNLOCK. *Rejected — a `:134-136`-style blanket early-return* (it sets `mouse_down = 0`, which makes the motion handler bail at `:192` before `dragging_pan` can arm, KILLING pan — violating "pan still works"). *Rejected — three scattered per-arm gates* (`:130`/`:159`/`:368`): they miss the reconnect arm (`:143-158`) and the marquee arm (`:170-171`), so locked edge-repoint and locked marquee-select would silently ship. *Edge case to note at implementation:* the top-of-block return preempts the conn-resolve at `:114`, so an in-flight connectOnClick cannot be click-COMPLETED while locked — it stays armed until lone-ESC cancels it (`src/flow_run.h:80`); acceptable, since connect-arming is itself suppressed under lock so this only affects a connection that was armed before lock was set.

*The render slot and themed chrome (the compose-order decision + the theme-foundation dependency).* `flow__controls` is called at `src/flow_render.h:336`, BETWEEN the minimap (`:336`) and the app-overlay hook (`:337`) — the approved "after minimap" slot. It draws AFTER the minimap so the two corner widgets compose predictably (if both target the same corner, controls draws on top — documented, not prevented), and BEFORE `on_overlay` so an app HUD can still draw over the controls if it wants (matching the minimap's under-app placement; the statusbar at `:341` is the only always-wins pass). The bar is a one-row `[+][-][⊞][🔒]`-style glyph run; the lock button renders a distinct glyph when `f->locked` (e.g. a filled vs open lock) so its state is visible. **Chrome color comes from the theme tokens introduced by #1 (theme-foundation):** the bar's fg/bg pass `f->theme.*` lookups, NOT the raw `FLOW_FG`/`FLOW_BG` literals — so it tracks light/dark mode. This is the one cross-package coupling: #1 lands the `flow_theme` struct + default preset (byte-identical to today's literals) FIRST, and `flow__controls` is written against `f->theme.*` from the start. If #1 has not landed when this is implemented, `flow__controls` uses `FLOW_FG`/`FLOW_BG` directly and a follow-up converts it — but the execution order pins #1 first precisely to avoid that. (Recorded under Depends on.)

*WHY lock is a flow-level bool, not a node-flag bit (the flag-ledger pin).* The approved flag ledger PINS that no node-flag bit is consumed this increment — bit 64u (`src/flow_model.h:2`, next-free after `FLOW_ANIMATED = 32u`) stays reserved for the deferred per-element interaction gates. A whole-canvas lock is naturally ONE bool on `struct flow` (`int locked;`), not a per-element flag: it gates the input path globally, every node/edge is equally locked, and there is no per-element variation to encode. A flag bit would also be ephemeral-but-per-element (the wrong granularity) and would burn the one reserved bit. So `locked` joins the transient scalar cluster beside the conn state (`src/flow_model.h:357`), calloc-zero default, and is queried via `flow_locked` rather than `flow_get_node(...)->flags`.

*Transient chrome: lock + controls config are never saved, never journaled (the JSON/undo pin).* `flow_save` emits no view/chrome state — the header already states "Ephemeral state (flags, selection, w/h, widgets) is NEVER emitted" (`src/flow_json.h:7-8`), and `locked`/`controls` are exactly that class of transient view state, the same as `minimap`/`statusbar`/`tick`/`key_hook_modal` (none of which `flow_save` touches). Nothing in this package journals (the undo stack records graph mutations, not chrome toggles — precedent: `tick`/modal state are not journaled). So `flow_save`/`flow_load` are byte-unaffected and the undo journal is untouched: a locked graph saved and reloaded comes back UNlocked (the app re-locks if it wants, the same convention as re-hiding or re-arming animation). No JSON or undo plumbing is added.

### 4. Node toolbar (selection-anchored action strip)  `[M]`  ·  id: `node-toolbar`

**Goal.** xyflow's `<NodeToolbar>` — a small action strip pinned just above the selected node, whose buttons act on that node — has no analogue in `flow`: there is no selection-anchored chrome and no way for an app to attach per-node actions to a click target. Today the only path to "do X to the selected node" is a global key binding (`flow_bind_key`, `src/flow_model.h:1346`) or a right-click context hook (`on_node_context`, `src/flow_model.h:288`) — neither is a visible, clickable, node-anchored affordance. This package adds one: the app hands the engine a **borrowed** array of `{label, fn, user}` actions via `flow_set_node_toolbar(f, actions, n)`; while exactly one node is selected (`flow_selected_count(f) == 1`, `src/flow_model.h:968`) the engine draws a one-row strip of labelled cells one row above that node's screen footprint, and a press landing on an action cell invokes that action's `fn` with the selected node's id — **consumed**, never falling through to the canvas. Anchoring reuses the existing screen-projection choke point `flow__node_footprint` (`src/flow_model.h:856`); hit-testing reuses the **widget seam** package 3 (`controls-bar`) installs at the top of the left-press classifier (`src/flow_input.h:105`). The toolbar is the SECOND rider on that one seam — controls is the first — proving the seam generalizes beyond a single fixed-position widget to a moving, selection-driven one.

**User value.** An app author gets xyflow's per-node quick-actions with one call and a static array — no key-binding ceremony, no context-menu hook, no manual hit-math. A button labelled `del` / `dup` / `↑` sits where the eye already is (on the selected node) and fires `fn(f, id, user)` with the node it belongs to, so the action body needs no global "which node is selected?" lookup. Because the array is borrowed (engine stores a pointer, never copies — see Design notes), the same `flow_toolbar_action[]` can be a file-scope `static const`; toggling the toolbar off is `flow_set_node_toolbar(f, NULL, 0)`. End users get a discoverable, mouse-reachable action surface that tracks the node under pan/zoom and stays a constant size in screen cells regardless of zoom (the LOD/position-only-scales contract, `src/flow_model.h:852-854`).

**API additions.**
```c
/* node toolbar (inc-7 #4): a selection-anchored action strip. SHARED action type —
   package 5 (edge-toolbar) reuses it verbatim (generic `int id` is a node OR edge id).
   Declared above the FLOW_IMPLEMENTATION guard (public). */
typedef struct {
  const char *label;                          /* borrowed cell text (app owns; engine never copies) */
  void (*fn)(flow_t *f, int id, void *user);  /* invoked with the SELECTED node id on an action-cell press */
  void *user;                                 /* opaque, passed through to fn */
} flow_toolbar_action;

/* Install the toolbar's action set. BORROWED: the engine stores `actions` as a pointer
   (NOT a heap copy) — the app must keep the array alive while it is set. NULL/0 = off.
   Shown only while flow_selected_count(f) == 1. Transient: never saved, never journaled.
   No flag bit consumed (toolbar visibility derives from selection; bit 64u stays reserved). */
void flow_set_node_toolbar(flow_t *f, const flow_toolbar_action *actions, int n);

/* On struct flow (FLOW_IMPLEMENTATION-guarded), in the widget-config cluster beside
   minimap/bg (src/flow_model.h:363-364). Value member, borrowed pointer, NO heap →
   inherits the single free(f) teardown (flow_free, src/flow_model.h:571-577) untouched.
   calloc default {NULL,0} = OFF, so NO flow_new line is added (src/flow_model.h:549-560). */
struct { const flow_toolbar_action *actions; int n; } node_toolbar;
```

**Design notes.**

*Conform to package 3's seam — do NOT invent it (the spine of this package).* Node-toolbar rides the interactive-widget hit-test that package 3 (`controls-bar`) introduces; that package owns the seam's exact symbol names and the precise consume mechanics. This package therefore names only the seam PROPERTIES it requires and adds its branch to whatever package 3 pins — the same discipline inc-6 used when `tick-autopan` (#8) deferred the arming-predicate name to `redraw-clock` (#4) and conformed by "adding its OR clause to whatever #4 names." The three seam properties node-toolbar depends on, all verified against the current input path: (1) the hit-test runs at the TOP of the `if (ev->button == 0)` block (`src/flow_input.h:105`), ABOVE the `conn_active` resolve (`:114`) and the `space_held` pan-arm (`:117`), so a toolbar-cell press can never be consumed first by an in-flight connection completion or a space-held pan; (2) a press the seam claims mirrors the handle-grab reset — `mouse_down=0; down_node=-1; drag_node=-1; dragging_pan=0; down_modsel=0; marquee_active=0; return;` (`src/flow_input.h:134-136`) — which makes the trailing RELEASE a clean no-op (the click branch gates on `mouse_down && !moved`, `src/flow_input.h:349`); (3) the test compares the raw screen cell `scr = {ev->x, ev->y}` (`src/flow_input.h:90`) against a SCREEN rect directly — NO `flow_to_world` projection (that wrapper, `src/flow_model.h:914`, is only for graph elements). Node-toolbar's contribution is one branch inside that seam: compute the toolbar's screen rect (the shared choke-point below), and if `scr` lands on action cell `k`, invoke `actions[k].fn` and consume. Package 3 ships the seam + the controls buttons; package 4 reuses it for a MOVING, selection-anchored widget — that reuse is the package's whole point.

*One shared layout choke-point — render and hit-test must never drift (the anti-drift discipline).* The toolbar's screen geometry is computed in exactly ONE place: a static `flow__node_toolbar_rect(flow_t *f, flow_rect *out_strip, int *out_cellx, int *out_n)` (or equivalent) that returns the strip's screen rect and the x-span of each action cell. The render pass (draw) and the input seam (hit-test) both call it — neither re-derives the math. This is the codebase's load-bearing convention: `flow__node_footprint` (`src/flow_model.h:856`) is shared by render (`src/flow_render.h:229`) and `flow_hit_node` so a node renders exactly where it is hittable; `flow__handle_screen` is shared by the handle marker loop and `flow_hit_handle` for the same reason (the choke-point comment, `src/flow_model.h:826-829`). A toolbar that computed its cells one way in render and another way in input would let clicks miss the visible buttons — the exact drift these choke points exist to forbid. **Placement trap (amalgamation order):** the choke-point must be visible to BOTH `flow_render` (module 11) and `flow_input` (module 13), so it lives in `flow_model.h` (module 4) — NOT in `flow_input.h` beside the seam. This is the identical trap inc-6 #6 documents for `flow__seq_len` ("MUST live in `flow_model.h` … the amalgamation order is `… flow_model … flow_input …`, so a flow_model caller cannot see a flow_input definition"). `flow_model.h` is the clean home; `tools/amalgamate.sh:5` ordering is the constraint.

*Anchoring math: footprint top, one row above, screen-space clamped.* The choke-point starts from `flow__node_footprint(f, n, flow__lod_for(f, f->view.zoom))` (`src/flow_model.h:856`) for the selected node `n` — giving the node's screen rect where it actually draws. The strip's baseline row is `footprint.y - 1` (one cell above the node's top border). The strip width is the sum of the rendered cell widths (`label` text length + a 1-cell pad/separator each), computed in SCREEN cells — it does NOT scale with zoom (only POSITION scales; the LOD note at `src/flow_model.h:852-854` is the contract). Horizontal origin: left-align to `footprint.x` by default, then clamp so the whole strip stays in `[0, cols)` (shift left if it would overrun the right edge). Vertical: if `footprint.y - 1 < 0` (node hugging the top, no room above), flip the strip to `footprint.y + footprint.h` (one row BELOW the node) — the "anchor above, flip below when no room" policy, stated so it is contractual, not incidental. Each action cell's x-span is recorded by the choke-point so the hit-test can map a `scr.x` to an action index.

*Offscreen / fully-scrolled-off behaviour (open question — recommended: clamp-and-keep).* When the selected node is panned partly or fully off-screen the strip's anchor can land outside `[0,cols)×[0,rows)`. Recommended v1 policy: clamp the strip to the nearest on-screen edge and keep it visible (an offscreen selected node still shows reachable actions — matches the "selection is global state, the toolbar is its chrome" model). The alternative — HIDE the toolbar when the anchor is offscreen — is recorded as the open fork; it is arguably tidier but strands the actions exactly when an app most wants "act on the thing I just selected and scrolled to find." Clamp-and-keep reuses the same edge-clamp idiom the minimap viewport rect already uses (`src/flow_render.h:66-67`). Either way the hit-test reads the SAME clamped rect from the choke-point, so visible == hittable holds.

*Why borrowed, not heap-owned (the struct-flow pin).* `flow_set_node_toolbar` stores `actions` as a pointer; the engine never `malloc`s or copies the array or its `label` strings. This is the dominant lifetime convention in `flow`: node/edge VTABLE registries are borrowed (`ntypes`/`etypes` are `const flow_node_type **`, `src/flow_model.h:330`), the string returned by `flow_node_type.label` (`src/flow_model.h:28`) is node-owned (borrowed by the engine, never copied), and the validator/hook `user` pointers (`src/flow_model.h:358-359`) are opaque borrows. It keeps the `node_toolbar` member a pure VALUE type, so it inherits the zero-cost teardown — the only heap-owning `struct flow` members are the dynamic arrays + clipboard + journal, freed by the single ordered `flow_free` (`src/flow_model.h:571-577`); a borrowed-pointer config field adds nothing there and survives `flow__graph_reset` by omission like every other non-graph config. The contract the API comment pins: **the app owns the array and must keep it alive while it is set** (a `static const flow_toolbar_action[]` is the canonical shape — the demos' callback structs are all file-scope). A heap-owning toolbar was rejected: it would force a `free` in `flow_free`, a survival decision across `flow__graph_reset`, and a deep-copy of the `label` strings — all over-scope for transient chrome and contrary to the approved struct-flow pin.

*Pass the node id (int), never a node pointer — and call `fn` LAST, then return.* The action signature is `fn(flow_t *f, int id, void *user)`, carrying the SELECTED node's id, not a `flow_node *`. This is deliberate: an action that deletes or adds nodes triggers `flow__grow`/realloc of `f->nodes`, dangling any cached pointer — the codebase's "re-fetch after a mutator, never cache across it" rule (the paste loop re-fetches `pn` after every `flow_add_node`, `src/flow_model.h:1196`; `flow_group` re-fetches after add, `:772`). Handing the action the id lets `fn`'s body `flow_get_node(f, id)` fresh. For the same reason the seam branch invokes `actions[k].fn(f, id, user)` and immediately `return`s — `fn` may itself call `flow_set_node_toolbar` (swap the action set), mutate the selection, or delete the node, so nothing after the call may touch toolbar state. (Reading `actions[k]` into locals BEFORE the call guards against an `fn` that frees/reassigns the borrowed array mid-dispatch.)

*Visibility = single selection, by selection NOT focus; node/edge exclusivity is free.* The strip shows iff `flow_selected_count(f) == 1`, with the subject `flow_selected_node(f)` (`src/flow_model.h:967-968`). Selection is the right trigger, not keyboard focus (`f->focus_node`, `src/flow_model.h:335`): focus is a navigation cursor (Tab ring) that exists even with nothing selected, whereas an action strip should appear on the thing the user committed to. Focus-as-trigger is the rejected alternative (it would flash the toolbar during Tab-cycling). Count `!= 1` (zero or multi-select) hides it — multi-node actions are out of scope for v1 (the selection-root machinery at `src/flow_model.h:711-718` is available if a future multi-target toolbar wants it). Node and edge selection are MUTUALLY EXCLUSIVE (`flow_select_edge` clears node selection, `src/flow_model.h:1008-1009`), so the node-toolbar and package 5's edge-toolbar are naturally never both visible — that exclusivity is a property of the selection model, NOT something either package gates explicitly. Recorded so the consistency pass confirms neither toolbar adds a redundant "is the other showing?" check.

*Compose order: after controls, under the app overlay (the z-slot decision) — and the matching hit-test order.* The toolbar pass slots at `src/flow_render.h:336-337` — AFTER the minimap and AFTER package 3's controls bar, and BEFORE the `on_overlay` app hook. Rationale: the app's `on_overlay` is the topmost engine-composed layer (xyflow `<Panel>` z-slot, drawn over a full-screen surface, `src/flow_render.h:337`), and an app HUD (e.g. inc-6's devtools HUD) must be able to draw OVER engine chrome — so the toolbar, like the minimap, sits just under it. The statusbar still wins over everything (drawn LAST, bottom row only, `src/flow_render.h:341`) but it is a single bottom row and cannot collide with a node-anchored strip. The toolbar draws into the same `flow_cellbuf` with the same primitives as the minimap; it does NOT build its own `flow_surface` for the app (it is engine chrome, not an app hook). **Render vs hit-test order must agree on overlap:** a selected node's strip can land over the fixed controls bar. Controls draws FIRST and the toolbar draws AFTER, so the toolbar wins the pixel (it is on top); the seam hit-test must therefore test the TOOLBAR cell before the controls cell so the click target matches what is rendered on top. Stated so the consistency pass confirms render-order and hit-order are inverses-consistent (topmost-rendered == first-hit), the same invariant the depth-ordered node draw/hit pair maintains.

*The shared action type spans to package 5 (recommendation-first scope call).* `flow_toolbar_action` carries a generic `int id`, not a `node` field — so package 5 (`edge-toolbar`) reuses the SAME type verbatim (`id` is then an edge id) and the SAME `fn` shape. ### 5. Edge toolbar (route-anchored action bar for the selected edge)  `[M]`  ·  id: `edge-toolbar`

**Goal.** Give the single selected edge a small floating action bar — the terminal analogue of xyflow's `EdgeLabelRenderer`/`EdgeToolbar` — anchored at the edge's route **midpoint** so a one-click `delete-edge` (and any app action) rides the connection itself instead of a menu. Today there is NO per-edge chrome: an edge is selectable (`flow_select_edge`/`flow_selected_edge`, `src/flow_model.h:1004-1011`) and removable (`flow_remove_edge`, `src/flow_model.h:1052`), but the only path to delete it is the keyboard `x`/Delete built-in over the whole selection (`src/flow_model.h:1393` — the `x` handler; Delete at `:1392`; both call `flow_delete_selection`, whose selected-edge loop is `src/flow_model.h:1127`) or an app-built right-click menu through `on_edge_context` (`src/flow_input.h:101-102`). The mechanism: a `flow__edge_toolbar` render pass composed in the widget tier (the seam **controls-bar** introduces at `src/flow_render.h:336`, before `on_overlay`) draws the configured actions one row above the route midpoint, and — reusing the **node-toolbar** package's shared `flow_toolbar_action` type and toolbar-provider pattern, and riding the **controls-bar** widget hit-test seam + `widgets[]` cache (top of the `button==0` press block, `src/flow_input.h:105`) — a click on an action cell fires that action and is consumed (never falls through to edge/pane/marquee). The anchor is the route's `label_anchor` (`flow_route.label_anchor`, `src/flow_cell.h:18`; set by every router — orthogonal `src/flow_route.h:62`, straight `src/flow_route.h:106`, degenerate `s==t` `src/flow_route.h:78`), recomputed each frame so it tracks the edge as endpoints move.

**User value.** An app author wires `flow_set_edge_toolbar(f, actions, n)` once and every selected edge gains a mouse-reachable [×] (or relabel/reroute) button parked on the wire — the standard "select an edge, act on it in place" affordance, with zero per-edge bookkeeping. The end user clicks the edge, sees the bar appear on the connection, clicks ×, the edge is gone — no menu hunt, no keyboard. Apps with no toolbar set are byte-identical to today (the config defaults NULL/0 via calloc), so the feature is invisible until armed. It composes with the **node-toolbar**: because node and edge selection are mutually exclusive in the model (`flow_select_edge` clears node selection, `src/flow_model.h:1009`), at most one toolbar kind is ever visible, with no overlap arbitration to author.

**Design notes.**

*Anchoring at `label_anchor`, recomputed every frame (the central seam, and why recompute not stash).* The route midpoint is `flow_route.label_anchor` (`src/flow_cell.h:18`), set by each router as the geometric middle of the path: orthogonal stores `p[n/2]` (`src/flow_route.h:62`), straight stores `((s.x+t.x)/2, (s.y+t.y)/2)` (`src/flow_route.h:106`), and the degenerate `s==t` case stores `s` (`src/flow_route.h:78`) — so the anchor is always defined for any visible edge. Critically, **the route is NOT stored on the edge** — `flow_render` builds `flow_route rt = {0}` per edge per frame (`src/flow_render.h:192`) and `free(rt.cells)` immediately after drawing (`src/flow_render.h:206`). There is no persisted `label_anchor` to read. So the toolbar pass must RECOMPUTE the route for the one selected edge: `flow__edge_screen_ends(f, e, &ss, &sp, &ts, &tp)` (`src/flow_render.h:189`, the shared helper that produces screen endpoints + facings) → `flow_edge_type_for(f, e->type[...])->route(ss, sp, ts, tp, &rt)` (`:193`) → `rt.label_anchor` is now in **screen-cell coords** (because `flow__edge_screen_ends` feeds screen endpoints into the router, the same reason the edge LABEL draws in screen coords at `:202-204`) → place the bar one row above `{rt.label_anchor.x, rt.label_anchor.y}` → `free(rt.cells)`. This is a single extra route computation per frame for the one selected edge — negligible, and it is THE established idiom (it is exactly what the edge loop already does). I REJECT the alternative of stashing `rt.label_anchor` into a `struct flow` field during the existing edge loop (option B): it couples two passes, and the edge loop early-`continue`s on an invisible edge or a missing endpoint (`src/flow_render.h:187,189`) — leaving a stale stashed anchor the toolbar pass would then render against. Recompute-on-demand has no stale-state failure mode and self-heals when the edge re-routes (drag an endpoint node, the midpoint moves, the next frame's recompute follows it — *the* property the task pins). Recompute also localizes ALL toolbar logic to one pass, so the edge loop at `:185-207` is byte-for-byte unchanged (goldens safe).

*The anchor must re-evaluate as the edge re-routes — and recompute delivers that for free.* The task underlines that the anchor "must recompute as the edge re-routes." Because the bar's screen position is derived fresh each frame from the live `flow__edge_screen_ends` + `route` output, it tracks every re-route source: a node drag (endpoints move → router recomputes the path → `label_anchor` moves), a pan/zoom (`flow__handle_screen` projection shifts), an endpoint reconnect (`flow_reconnect_edge`, `src/flow_model.h:1012`). No invalidation hook, no dirty flag — the determinism is that render is a pure function of model+view, and the anchor is part of that function. The one subtlety to PIN: the toolbar pass must use the SAME `flow__edge_screen_ends`/`route` call the draw loop uses (not re-derive endpoints by hand), or a divergence would float the bar off the wire — the controls/node-toolbar shared-helper discipline (geometry choke points, `flow_node_rect_abs`+`flow_to_screen` for nodes; `flow__edge_screen_ends`+`route` for edges) exists precisely to forbid that re-derivation.

*Edge-selection read, and mutual exclusivity with node selection (the model contract).* Selection is the `FLOW_SELECTED` flag on `flow_edge.flags` (`src/flow_model.h:2,19`), read via `flow_selected_edge` (`src/flow_model.h:1004-1007`, "first selected edge id or -1"). The toolbar's visibility gate is exactly `f->edge_toolbar.actions && f->edge_toolbar.n > 0 && flow_selected_edge(f) != -1`. There is NO multi-edge case to consider: `flow_select_edge(f, id, additive)` with `!additive` calls `flow_clear_selection` which clears node AND other edge selection (`src/flow_model.h:1008-1009`), and even the additive paths in the input layer (`src/flow_input.h:340,371`) route single edge ids — the engine never builds a multi-edge selection set the way marquee builds multi-node sets. So "single selected edge" is the natural and only state; the gate is `!= -1`, not a count. And because node-vs-edge selection is mutually exclusive (the same `flow_clear_selection` call at `:1009`), when an edge is selected NO node is selected — so the node toolbar (which gates on `flow_selected_node(f) != -1` / its own single-node rule) is simultaneously hidden. The two toolbars are mutually exclusive by the model's own invariant; this package adds no arbitration code, it just relies on the existing clear-on-select.

*Shared widget-seam dispatch — one hit-test surface, two providers (the reuse story).* This package adds NO new input-path branch and NO new dispatch mechanism. The **controls-bar** package introduces BOTH the single widget hit-test at the top of the left-press block (`src/flow_input.h:105`, above the conn-active resolve at `:114`, space-pan at `:117`, and the handle→edge→node trio at `:124-159`) AND the shared widget hit-rect cache — the `widgets[16]` / `nwidgets` value array on `struct flow` (modeled on `keys[32]`, `src/flow_model.h:365`, recomputed each render — NO heap). The **node-toolbar** package (#4) adds the first toolbar provider on that cache via its shared layout choke-point `flow__node_toolbar_rect` (the render/hit-test single-source for a toolbar's cell rects). The edge toolbar is the SECOND TOOLBAR PROVIDER on that one surface: its render pass computes its cell rects (route-midpoint anchor, vs the node toolbar's node-rect-derived anchor via `flow__node_toolbar_rect`) and pushes them into the SAME `widgets[]` cache that controls-bar (#3) defines, each entry carrying enough to route the click. The `:105` seam matches the clicked cell against the cache and fires the owning action — for an edge-toolbar cell it resolves `actions[k].on_click(f, flow_selected_edge(f), user)`. (This requires #3's cache entry to carry an owner/provider discriminator beyond the `{x,y,w,h,action}` controls-only shape — flagged to #3 as the one shared-cache widening the toolbars need; #4 is the first to exercise it, #5 the second.) The press-consume follows the established widget pattern: on a hit, set `mouse_down=0; down_node=-1; drag_node=-1; dragging_pan=0; down_modsel=0; marquee_active=0; return` (the handle-grab reset shape at `src/flow_input.h:134-136`), which makes the trailing RELEASE click branch (`src/flow_input.h:368-386`, edge-body select at `:371`/`on_edge_click` at `:373`) a clean no-op — so a click on the × cell does NOT also fire `on_edge_click` or re-select. **The only edge-toolbar-specific code is: (1) the config field + setter, (2) the render pass that supplies the midpoint anchor and the edge id, and (3) the dispatch case that reads `flow_selected_edge` as the action id.** Everything else (the seam, the cache, the renderer, the consume) is shared with controls + node-toolbar.

*Compose order — within the widget tier, AND between the two toolbars.* In the render compose order, the toolbar pass slots at `src/flow_render.h:336` (with the minimap, before `on_overlay` at `:337`), so app overlays still win — consistent with the minimap precedent and the node-toolbar's placement. The bar is drawn AFTER edges (`:185-207`), nodes (`:215-241`), handles (`:245-256`), the focus ring (`:264-277`), the connection preview (`:281-297`), the marquee (`:304-318`), and the helper guides (`:325-334`) — so it sits on top of the wire it annotates and is never occluded by graph content. Within the two toolbar providers, the order is irrelevant to pixels because they are mutually exclusive (a frame shows at most one), but to keep the dispatch cache unambiguous the node toolbar fills its cells first, then the edge toolbar — only one provider ever populates the cache in a given frame. PIN: the bar's anchor row is `label_anchor.y - 1` (one row ABOVE the midpoint, so the wire stays visible beneath the bar); if the anchor sits in row 0 the bar clamps to row 0 (draw on the midpoint cell rather than off-buffer), mirroring the helper/marquee off-screen clamp convention (the `continue` guards at `src/flow_render.h:327,332`). Off-buffer columns are clamped per-cell by `flow_cellbuf_put` (it bounds-checks), so a midpoint near the right edge simply truncates the bar — no special casing.

*Why a borrowed-pointer value struct, not a heap-owned action list (the ownership decision).* The config mirrors `flow_set_minimap`'s value-struct precedent exactly (`struct { int enabled, w, h; flow_corner corner; } minimap;`, `src/flow_model.h:363`; setter `:998-1000`): `flow_set_edge_toolbar` stores a BORROWED `const flow_toolbar_action *actions` + `int n` into a small value member on `struct flow`. The actions array is caller-owned, static-lifetime (the same contract as a `flow_node_type` table or `flow_callbacks`), so the engine never copies or frees it — inheriting the zero-cost teardown the recon flags for every config member (the only heap owners are nodes/edges/clipboard/journal; `flow_free` at `src/flow_model.h:571-577` is untouched). This is also why nothing changes in `flow__graph_reset` (`src/flow_model.h:583-592`): a config pointer is chrome, not graph state, and survives a reset like the minimap/bg config does. A heap-owned, engine-copied list was rejected: it would force a `free` in `flow_free` and a copy in the setter for zero benefit — the action table is fixed at app-init like every other vtable in the library.

*Flag ledger: NO bit consumed (the pinned constraint).* This package adds NO node/edge flag bit. Edge selection already lives in `FLOW_SELECTED` (bit 1u, `src/flow_model.h:2`), and the toolbar config + hit-rect cache are flow-level state on `struct flow`. The flag enum currently runs through `FLOW_ANIMATED = 32u` (`src/flow_model.h:2`), so bit 64u stays free/reserved for the deferred per-element interaction gates (the increment-7 flag-ledger pin). The toolbar is gated by `flow_selected_edge(f) != -1` plus the config pointer — no per-edge "has-toolbar" marker, no per-edge state at all.

*JSON / undo: transient chrome.* The edge-toolbar config (`actions`/`n`) and the hit-rect cache are TRANSIENT view/chrome state: never emitted by `flow_save` (flags AND widgets are already in the never-emitted list, `src/flow_json.h:7-8`; config pointers were never in the schema), never journaled (precedent: minimap/bg/statusbar config, and inc-6's tick/modal state, are all un-journaled and un-saved).
