# flow — architecture

How `flow` is built and laid out, for contributors and curious embedders. This is a
conceptual guide; for per-function detail see [`docs/API.md`](API.md), and for a copy-paste
start see the [README](../README.md).

`flow` is a single-header C library for interactive node-graph editors in the terminal — a
C/terminal analog of [xyflow](https://github.com/xyflow/xyflow) (React Flow). The shipping
artifact is one file, `flow.h`. This document explains where that file comes from, how its
15 source modules stack up, and where the portable core ends and the POSIX terminal begins.

## The amalgamation model

`flow.h` is **generated**, not hand-written. The real source lives in `src/*.h` fragments;
`tools/amalgamate.sh` concatenates them, in a fixed dependency order, into the single
committed header.

```sh
sh tools/amalgamate.sh   # src/*.h  ->  flow.h   (or just `make`)
```

`flow.h` is a **build product**. Its first line says so:

```c
/* flow.h — GENERATED by tools/amalgamate.sh. Do not edit; edit src/ instead. */
```

It is committed to the repo (so consumers can vendor a single file without running the
build), but every change starts in `src/`. Editing `flow.h` directly is always wrong: the
next `make` overwrites it, and CI rejects the drift (see [CI gates](#ci-gates) below). The
edit/build/verify loop for contributors is in the README **Developing** section.

The wrapper that `amalgamate.sh` emits is minimal: an outer `#ifndef FLOW_H` include guard,
then each `src/$m.h` verbatim under a banner comment, then the close of the `extern "C"`
block that `flow_head.h` opens (so the header is C++-consumable). The module list — the
`modules=` line in `tools/amalgamate.sh` — is **dependency-ordered**; reordering it breaks
the build, because later modules call into earlier ones.

### The `FLOW_IMPLEMENTATION` decl/impl split

Like other [stb-style](https://github.com/nothings/stb) single headers, `flow.h` is two
headers in one file, separated by a macro:

- **Declarations** (types, enums, function prototypes) are **always visible** — every TU
  that includes `flow.h` sees them.
- **Definitions** (function bodies, file-scope tables, `static` helpers) live behind
  `#ifdef FLOW_IMPLEMENTATION` and compile in **exactly one** translation unit.

```c
/* In exactly ONE .c file: */
#define FLOW_IMPLEMENTATION
#include "flow.h"

/* Everywhere else: declarations only. */
#include "flow.h"
```

Each `src/*.h` module carries its own `#ifdef FLOW_IMPLEMENTATION` block, so the split is
preserved module-by-module after amalgamation rather than being one giant guard at the end.
Defining the macro in two TUs gives duplicate-symbol link errors — define it once.

## Module map

The 15 modules in `amalgamate.sh` `modules=` order, each with a one-line "owns"
description derived from its top-of-file comment. Dependencies point **upward**: a module
may call anything above it, never below.

| # | Module | Owns |
|---|--------|------|
| 1 | `src/flow_head.h` | License, version macros, libc + POSIX includes, the `extern "C"` open, allocator hooks (`FLOW_MALLOC`/`FLOW_CALLOC`/`FLOW_FREE`). |
| 2 | `src/flow_geom.h` | Geometry — pure, no I/O: `flow_pt`/`flow_rect`/`flow_viewport`, `flow_project`/`flow_unproject` world↔screen mapping. |
| 3 | `src/flow_cell.h` | Cells, the surface buffer, box-drawing, UTF-8 encode, and the damage diff (`flow_cell`, `flow_diff_emit`); also `flow_theme` + `flow_color_mode`. |
| 4 | `src/flow_model.h` | The engine: nodes, edges, custom-type vtables, flags, transforms, bounds, hit-test. The `flow_t` struct and most mutators. |
| 5 | `src/flow_undo.h` | Undo/redo — the appliers that replay the command journal in reverse via inverses (recording primitives are pushed from `flow_model`). |
| 6 | `src/flow_query.h` | Read-only graph traversal: `flow_incomers`/`flow_outgoers`/connected-edge queries over `f->edges`. |
| 7 | `src/flow_view.h` | Viewport mutators: pointer-centered zoom, zoom limits, recenter, level-of-detail (`flow_set_zoom`, `flow_lod_for_zoom`). |
| 8 | `src/flow_layout.h` | Auto-layout — a stateless model→positions transform driving only public mutators (`flow_layout`); force-directed and layered. |
| 9 | `src/flow_route.h` | Edge routing: orthogonal step router and straight router emitting path-connectivity glyphs (`flow_route_orthogonal`/`flow_route_straight`). |
| 10 | `src/flow_types.h` | The built-in node/edge types: `flow_default_node_type`, `flow_group_node_type`, default handles, `flow_register_defaults`. |
| 11 | `src/flow_render.h` | Composition: render edges + nodes (+ background grid + chrome) into a `flow_cell` buffer (`flow_render`). Reads theme tokens. |
| 12 | `src/flow_json.h` | JSON persistence — dependency-free `flow_save`/`flow_load` of durable structure only. The sole untrusted-input parser. |
| 13 | `src/flow_input.h` | Mouse input: the SGR-1006 (`\x1b[<…M/m`) parser plus drag/pan/connect interaction. A pure parser — no syscalls. |
| 14 | `src/flow_term.h` | Terminal control: raw mode, alt-screen, size query, signal/crash restore handlers. POSIX-only (`termios`/`ioctl`/`poll`). |
| 15 | `src/flow_run.h` | Present (diff + flush), the embed primitives (`flow_render_diff`/`flow_present`), key feed (`flow_feed`), and the `flow_run` loop. |

New `src/*.h` modules must be added to the `modules=` line at the right dependency
position; new test suites to `TESTS=` in the `Makefile`.

## The portable core / POSIX terminal boundary

`flow` is split by what it touches. Most modules are **pure, portable C** — they compute on
in-memory structs and never call a syscall. A thin shell at the bottom (`flow_term`,
`flow_run`) **owns the TTY**.

| Layer | Modules | Touches |
|-------|---------|---------|
| **Split (header)** | `flow_head` | Includes only. Its libc declarations (`stdint`/`stddef`/`stdio`/`stdlib`/`string`/`math`, `assert`) are unconditional and portable; its POSIX includes (`termios.h`, `unistd.h`, `sys/ioctl.h`, `signal.h`, `poll.h`, `errno.h`) are emitted **only** under `#ifdef FLOW_IMPLEMENTATION`. |
| **Portable core** | `flow_geom`, `flow_cell`, `flow_model`, `flow_undo`, `flow_query`, `flow_view`, `flow_layout`, `flow_route`, `flow_types`, `flow_render`, `flow_json` | Memory only. No terminal, no `read`/`write`, no `time()`. |
| **Pure parser** | `flow_input` | Memory only — parses byte strings into events. Driven by the run-loop, but callable standalone. |
| **POSIX terminal** | `flow_term`, `flow_run` | `termios` raw mode, `poll(stdin)`, `write(stdout)`, `ioctl` size, signal handlers. |

`flow_head` straddles the boundary: a declarations-only (non-implementation) TU pulls **no** POSIX headers — the six `#ifdef FLOW_IMPLEMENTATION` includes never fire — so the header is portable on that side. The implementation TU is the opposite: those POSIX includes are hoisted unconditionally on *every* platform (there is no `_WIN32` guard), so the single TU that `#define`s `FLOW_IMPLEMENTATION` is POSIX-bound regardless of host. That is the concrete reason the implementation cannot compile on a non-POSIX host like Windows — which is the limit behind the run-loop note at the end of [Terminal honesty](#terminal-honesty).

`flow_input` deserves the separate row: it is a pure parser (a byte buffer in, mouse/key
events out), so it belongs to the portable side. What makes the *interactive* path
POSIX-only is `flow_run` driving that parser from a real `poll(stdin)` loop and flushing to
`stdout`, plus the `flow_term_*` mode-switching it brackets.

### The embedding seam

The boundary is also a public API seam: you can drive the entire model and produce frames
without any terminal at all. The README **Embedding** section names the two calls that do
it:

- **`flow_feed(f, bytes, n)`** — hand the model input bytes from any source (your own
  `read()`, a socket, a scripted replay). Same key/mouse parser `flow_run` uses.
- **`flow_render_diff(f)`** — render the model, diff it against the previous frame, advance
  the front buffer, and **return** a malloc'd escape string (absolute-positioned CSI/SGR)
  that you write to whatever fd you own, then `free()`. Returns `""` (empty, non-NULL) when
  nothing changed.

`flow_present(f)` is exactly `flow_render_diff` plus `fputs(stdout)` + `fflush` — the only
difference is who owns the fd. For the full back buffer instead of a diff, call
`flow_render(f, cells, cols, rows)` and read the `flow_cell` grid yourself. A complete
TTY-free program is in [`examples/embed_headless.c`](../examples/embed_headless.c)
(`make examples`).

So: a host with its own event loop never calls `flow_run`. It owns its I/O and calls
`flow_feed` + `flow_render_diff` (or `flow_render`). Only `flow_run` / `flow_present` /
`flow_term_*` require a POSIX terminal.

## Data flow

Two directions cross the seam: input bytes flow **in** to mutate the model, and the model
flows **out** through render and a damage diff to escape sequences.

### Input: bytes → events → model mutation

```
input bytes ──► flow_feed ──► key dispatch  ──► model mutators
(stdin / socket / replay)  ├─► flow_input    (move/select/connect/zoom/undo…)
                           │   (SGR-1006 mouse parse)
                           └─► CSI key parse
```

`flow_feed` (in `src/flow_run.h`) scans the byte buffer: mouse CSI sequences route to the
`src/flow_input.h` parser, which turns `\x1b[<…M/m` into a `flow_mouse_event` and applies
the drag/pan/connect interaction; key and escape sequences route to keyboard command
dispatch (built-ins plus a binding registry). Either way the result is a **model
mutation** through the same public mutators an embedder would call directly. Input never
renders — it only changes state.

### Output: model → cells → damage diff → escapes

```
model ──► flow_render ──► flow_cell back buffer ──► flow_diff_emit ──► CSI/SGR escapes
                          (cols × rows grid)        (vs. front buffer)   ("" if unchanged)
                                                          │
                                              flow_render_diff advances front = back
                                                          │
                                                  flow_present → stdout
```

`flow_render` composes the background grid, edges, nodes, and chrome into a flat
`cols × rows` array of `flow_cell` (each cell: a UTF-8 codepoint plus `fg`/`bg`/`attr`).
`flow_diff_emit` walks the new (back) buffer against the retained **front** buffer
(`f->front`) and emits an escape only for cells that changed: an absolute cursor move
`\x1b[<row>;<col>H`, an SGR style reset+set when style differs from the last run, then the
glyph. `flow_render_diff` wraps this, then `memcpy`s back over front to advance the frame.
Unchanged screens cost an empty string and zero writes. This is the standard
damage-tracking redraw — only dirty cells hit the wire.

The interactive `flow_run` loop ties it together: `poll(stdin)` with a timeout that is the
redraw interval `tick_ms` when animation is armed and `-1` (block forever) when idle — so a
static graph causes zero wakeups. On readable input it `flow_feed`s then `flow_present`s; on
timeout it advances the animation clock, replays in-flight auto-pan, and presents.

## The animation clock

Animation (marching-ants edges, auto-pan momentum) runs on a **deterministic integer
clock**, never wall-clock time. `flow_tick(f)` does exactly `++f->tick`; `flow_ticks(f)`
reads it. Render derives dash phase from `tick % FLOW_DASH_PERIOD`. The run-loop calls
`flow_tick` on each poll timeout, but the tick is the *only* animation input to the
model/render path — `time()` never enters it. This is what makes animated frames
snapshot-testable: a test advances the clock by hand and captures a byte-exact frame, with
no flakiness from real time.

## Test substrate

The headless test suite lives under `tests/` — **39 suites** (the `TESTS=` list in the
`Makefile`), each a standalone `tests/test_*.c` that `#define`s `FLOW_IMPLEMENTATION`,
includes `flow.h`, exercises the library, and exits non-zero on failure. They are
**headless**: no terminal, no `flow_run`. They drive the model and the portable render path
directly, which is exactly why the embedding seam is testable.

```sh
make test    # build + run all 39 suites; non-zero exit on any failure
```

Rendering and persistence are pinned by **byte-exact snapshot goldens** in
`tests/snapshots/*.txt` — committed UTF-8 captures of rendered frames and JSON output. A
render or model change that shifts output fails its snapshot test until the golden is
regenerated (delete the stale `.txt`, rerun to recapture). Combined with the integer
animation clock, animated frames are fully deterministic goldens too (e.g.
`animated_tick0.txt`, `clock_tick0.txt`).

### CI gates

`.github/workflows/ci.yml` runs five jobs:

| Job | Gate |
|-----|------|
| **build-test** | Builds lib + demos + examples and runs all suites on a matrix — `{ubuntu, gcc}`, `{ubuntu, clang}`, `{macos, clang}` — with `-Werror` on top of `-Wall -Wextra`. |
| **check-amalgamation** | Regenerates `flow.h` from `src/` and `git diff --exit-code flow.h`. The crown-jewel gate: a PR that edits `src/` but forgets to regenerate is rejected before it ships a stale header. |
| **sanitizers** | Builds and runs every suite under ASan + UBSan (`-fsanitize=address,undefined -fno-sanitize-recover=all`), fail-accumulating across suites. |
| **cpp** | `make cpp`: proves `flow.h` compiles + runs as a C++ TU, and a C-compiled `flow.o` links against a C++ caller (the `extern "C"` guard). |
| **lint** | `clang-tidy` on a whole-library TU — **advisory** (`continue-on-error`), honoring the committed `.clang-tidy`. |

Mark **check-amalgamation** a required check in branch protection — it is the gate that
keeps the generated header honest. The local equivalents (the same ASan/UBSan loop, the
clang-tidy invocation) are the project's pre-commit verification, runnable via the
`flow-verify` workflow.

## Terminal honesty

`flow` renders in cells, and the architecture inherits the terminal's limits — it does not
hide them:

- **Edges are box-drawing corners**, not bezier curves. The router (`src/flow_route.h`)
  emits orthogonal step paths and straight runs from `┌ ┐ └ ┘ ─ │`-class glyphs.
- **One glyph = one cell.** There is no sub-cell magnification: zoom changes node *spacing*
  and level-of-detail (`flow_lod_for_zoom`), not glyph size.
- **Labels are measured one codepoint per cell.** Wide/CJK/emoji glyphs occupy two terminal
  columns but are counted as one, so they misalign — `flow_diff_emit` and the measure path
  assume monospace single-width.
- **The interactive run-loop is POSIX-only.** Windows and non-TTY hosts use the portable
  core + the embedding seam, not `flow_run`.

## Security surface

The portable core operates on in-memory structs the embedder constructs. The one entry that
parses **untrusted bytes** is `flow_load` (`src/flow_json.h`) — a hand-rolled,
dependency-free JSON reader. Treat any externally-sourced `.flow.json` as untrusted input to
that function; everything else in the API takes already-validated in-process data.
